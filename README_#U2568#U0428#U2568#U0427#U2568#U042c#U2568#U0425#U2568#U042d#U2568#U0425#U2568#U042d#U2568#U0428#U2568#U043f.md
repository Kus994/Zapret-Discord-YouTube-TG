# Доработки KUS Pro — что изменено и зачем

Эти файлы заменяют одноимённые файлы в корне проекта (структура папок такая же:
`main.py`, `page_cleanup.py`, `page_updates.py`, `downloader.py` — в корень,
`cleanup.py` — в папку `modules/`).

Затронуты только нейтральные модули (очистка системы, загрузчик файлов,
обновления Windows, точка входа). Код, относящийся к Zapret
(`modules/extra.py`, `page_zapret.py`, папка `zapret/`), не менялся.

## 1. page_cleanup.py + modules/cleanup.py — баг с нерабочими чекбоксами

**Было:** на странице «Очистка» шесть чекбоксов («Temp пользователя»,
«Системный Temp», «Кэш обновлений», «Temp на всех дисках», «Prefetch»,
«Корзина»), но реально читались только три из них. Если пользователь
снимал галку «Корзина» или «Temp пользователя» — это никак не влияло на
результат: корзина и оба Temp всё равно очищались.

**Стало:** все шесть пунктов реально работают. `clean_temp_folders()` и
`run_full_cleanup()` в `modules/cleanup.py` получили параметры
`include_user_temp`, `include_system_temp`, `include_recycle_bin`, и
`page_cleanup.py` прокидывает состояние всех галок. Также добавлена
проверка: если ни один пункт не выбран, очистка не запускается, а в лог
выводится предупреждение.

Заодно убран избыточный самоимпорт функции `clean_prefetch` внутри
собственного модуля `cleanup.py`, и параметр-флаг переименован в
`clean_prefetch_flag`, чтобы не путаться с именем функции `clean_prefetch`.

## 2. downloader.py — защита от Zip Slip (path traversal)

**Было:** при распаковке скачанного `.zip`-архива (используется и для
TgWsProxy, и для обновления Zapret) имена файлов внутри архива не
проверялись. Архив с вредоносным путём вида `../../evil.txt` мог
записать файл за пределами целевой папки распаковки.

**Стало:** `_extract_zip()` резолвит итоговый путь каждого файла и
сверяет, что он действительно лежит внутри целевой директории; записи,
пытающиеся выйти за её пределы, пропускаются. Проверено на синтетическом
архиве с `../../` — вредоносные файлы больше не выходят за пределы
`dest_dir`, легитимные файлы распаковываются как обычно.

## 3. main.py — применение настройки «Запускать свёрнутым в трей»

**Было:** на странице «Настройки» можно было включить «Запускать
свёрнутым в трей», значение сохранялось в `config.json`, но при
следующем запуске приложения никто его не читал — окно всегда
открывалось на экране.

**Стало:** `main.py` читает `start_minimized` из настроек перед показом
окна и, если включено, сразу скрывает окно в трей вместо показа на
экране (логика идентична нажатию кнопки «В трей»).

*(Остальные настройки — `autostart_zapret`, `autostart_tg_proxy`,
`tg_proxy_port` — не трогал: первая относится к Zapret, а вторая и
третья завязаны на конфигурацию стороннего `.exe` TgWsProxy, в которую
без документации лучше не лезть вслепую, чтобы не сломать рабочий
функционал.)*

## 4. page_updates.py — добавлен интерфейс для уже существующих функций

В `modules/updates.py` были полностью готовые, но никак не подключённые
к интерфейсу функции:
- `search_driver_updates` — поиск обновлений драйверов;
- `register_scheduled_task` / `unregister_scheduled_task` — создание и
  удаление задачи в Планировщике Windows для автоматической периодической
  проверки обновлений.

Добавил на страницу «Обновления»:
- кнопку «🖥 Обновления драйверов»;
- блок «Автоматическая проверка по расписанию» с выбором периодичности
  (ежедневно / еженедельно / при входе в систему) и времени запуска,
  и кнопками включения/отключения авто-проверки.

## 5. page_monitor.py — обновлён внешний вид страницы «Мониторинг»

**Было:** обычные горизонтальные прогресс-бары для CPU, RAM и дисков —
функционально, но визуально скучно по сравнению с остальным приложением.

**Стало:** страница переоформлена в стиле финансовых/аналитических
дашбордов (круговые гейджи, мини-графики):

- **Кольцевые индикаторы** CPU и RAM — круговой прогресс вместо линейного,
  с цветом по уровню загрузки (зелёный / жёлтый / красный) и процентом
  в центре кольца. Нарисованы вручную через `QPainter`, без сторонних
  библиотек.
- **Мини bar-chart по ядрам CPU** — рядом с кольцом CPU показывается
  текущая загрузка каждого ядра отдельными цветными столбиками.
- **Sparkline истории нагрузки CPU** — график-«область» за последние
  60 замеров (около 1.5 минуты при шаге обновления 1.5 с), показывает
  тренд нагрузки во времени.
- **Диски** — прогресс-бары с градиентной заливкой (тёмный → яркий
  оттенок того же цвета) вместо однотонной полосы; строки дисков
  создаются динамически и скрываются, если диск временно недоступен
  (например, отключили внешний накопитель), без пересборки всего layout.

Все данные берутся из уже существующего `modules/monitor.get_snapshot()` —
сам модуль данных не менялся, изменилась только визуализация.

## 6. modules/game_mode.py + page_game_mode.py + main_window.py — новая страница «Игровой режим»

**Источник:** два пользовательских `.bat`-скрипта («Оптимизация.bat» и
«Возврат.bat») из архива со скриптами — закрывали/запускали набор
фоновых программ (Wallpaper Engine, PowerToys, Windhawk, QuickLook,
AnVir, Everything, AnyDesk), переключали прозрачность окон и
авто-скрытие панели задач.

**Что сделано:** логика портирована в Python и встроена в приложение
как полноценная страница «🎮 Игровой режим» в сайдбаре, с доработками:

- **Пути не зашиты в код.** В исходных `.bat` пути были прибиты к
  конкретному диску и профилю пользователя (`F:\SteamLibrary\...`,
  `C:\Users\ErlP\...`) — у любого другого человека скрипт просто не
  сработал бы. Теперь это поля «Путь к .exe» с кнопкой «Обзор» в
  интерфейсе; список приложений и пути сохраняются в `config.json`
  (секция `game_mode`).
- **Список приложений настраиваемый и включаемый.** Каждое приложение
  можно временно отключить чекбоксом без удаления из списка.
- **Закрытие процессов через `psutil`**, а не `taskkill` — точно
  знаем, что реально было закрыто, что не найдено, а что не удалось
  закрыть (например, нет прав), и пишем это в лог построчно, а не
  угадываем по `%errorlevel%`, как было в `.bat` (там `%errorlevel%`
  после `taskkill /f /t /im X >nul 2>&1` не всегда отражал, был ли
  процесс на самом деле найден — отсюда в скрипте встречались неверные
  сообщения вида «AnVir закрыт» при попытке закрыть Everything).
- **Реестр меняется напрямую через `winreg`**, без вызова
  `cmd`/`powershell`. Кстати, в исходном `Возврат.bat`/`Оптимизация.bat`
  PowerShell-команда для панели задач (`$v=2; Set-ItemProperty ...`)
  записывала в `StuckRects3\Settings` целое число вместо ожидаемого
  48-байтного бинарного значения — это могло привести к ошибке
  PowerShell или повреждению настроек панели задач при выполнении
  «как есть». В новой реализации сохраняется только нужный бит
  авто-скрытия внутри существующей бинарной структуры, остальные байты
  не трогаются.
- **«Обычный режим» не пытается убить процессы, которые скрипт и
  раньше не должен был трогать** (Phone Link/RuntimeBroker закрываются
  только при входе в игровой режим, не запускаются обратно — ровно как
  было задумано в исходном `.bat`).

Страница регистрируется в `main_window.py` между «Мониторинг» и
«Обновления» в общем списке `_pages_def`.

**Проверка:** логика `close_apps` / `start_apps` (успешное закрытие,
процесс не найден, ошибка доступа при закрытии, отсутствующий/неверный
путь при запуске) и логика страницы (загрузка/сохранение конфигурации,
фильтрация включённых приложений, сквозной вызов от UI до модуля)
протестированы через мок `psutil`/PyQt5 — все сценарии прошли успешно.
Изменения реестра (`winreg`, перезапуск `explorer.exe`) специфичны для
Windows и не могут быть протестированы в этой среде — рекомендую
проверить на реальной машине, что прозрачность и авто-скрытие панели
задач переключаются как ожидается.



## 7. Новые функции 2026 года

### 7.1. modules/network.py + page_network.py — живая скорость сети и активность по процессам

- **Текущая скорость сети** — реальный замер через дельту счётчиков
  `psutil.net_io_counters()` (без сторонних спидтест-сервисов),
  обновляется в отдельном фоновом потоке `_SpeedThread` каждую секунду
  и выводится в виде двух «бейджей» ⬇/⬆ с возможностью поставить на
  пауза/продолжить. Поток не использует общий `_run_worker` страницы —
  иначе обновление каждую секунду блокировало бы кнопки «Обновить» и
  «Сброс TCP/IP».
- **Топ процессов по сетевой активности** — список процессов с
  наибольшим числом активных соединений (через `psutil.net_connections`).
  Это не байты/сек по процессу (это требует перехвата пакетов с правами
  администратора и отдельного драйвера) — а количество установленных
  соединений, что даёт быстрый и честный ответ на вопрос «что вообще
  держит сеть прямо сейчас» без сторонних снифферов.

### 7.2. modules/cleanup.py + page_cleanup.py — кэш браузеров и поиск дублей

- **Очистка кэша браузеров** (Chrome, Edge, Brave, Opera, Yandex
  Browser, Firefox) — удаляется только `Cache`/`Code Cache`/`GPUCache`
  (для Firefox — `cache2`), пароли/история/закладки не трогаются.
  «Найти браузеры» сканирует реальные профили на диске и показывает
  чекбоксы только для тех браузеров, что действительно установлены —
  не захардкоженный список всех потенциально поддерживаемых.
- **Поиск дублей файлов** — указываешь папку, утилита группирует файлы
  сначала по размеру (дёшево), затем хеширует SHA-256 только
  кандидатов с совпадающим размером (не весь диск целиком). Результат —
  таблица групп с отметкой «оставить» (●) на первом файле каждой
  группы; кнопка «Удалить отмеченные» убирает остальные после
  подтверждения. Удаление — безвозвратное (не в корзину), как и у
  остальных операций очистки в приложении.

### 7.3. modules/security.py + page_security.py — новая страница «Безопасность»

Аудит автозагрузки Windows: реестр `HKCU/HKLM ...\Run` + папка
автозагрузки (`shell:startup`). Это **не антивирус** — модуль не
определяет вредоносность, а показывает простые эвристические
предупреждения (⚠), которые стоит проверить самому:
- путь к файлу не существует;
- запуск напрямую из временной папки (`Temp`/`tmp`);
- запуск через закодированную команду PowerShell
  (`powershell -EncodedCommand ...`).

Удаление записи из таблицы убирает её только из автозагрузки
(значение в реестре или ярлык) — саму программу это не трогает.

### 7.4. Игровой режим — таймер автовозврата и глобальная горячая клавиша

- **Таймер автовозврата**: чекбокс + поле минут на странице «Игровой
  режим» — если включить, через указанное время после активации
  игрового режима утилита сама вернёт обычный режим (на случай если
  забыли выключить руками после игры/стрима).
- **Глобальная горячая клавиша** (например `Ctrl+Alt+G`) — переключает
  игровой режим даже когда окно приложения свёрнуто или не в фокусе.
  Реализовано через нативный `RegisterHotKey`/`UnregisterHotKey`
  (`modules/hotkeys.py`) и перехват `WM_HOTKEY` в
  `MainWindow.nativeEvent` — без сторонних библиотек типа `keyboard`.
  Хоткей переключается без модального диалога подтверждения (диалог
  посреди использования хоткея сделал бы его бесполезным), но обычные
  кнопки на странице по-прежнему спрашивают подтверждение. Требует
  перезапуска приложения после изменения комбинации в настройках.
  На платформах кроме Windows код безопасно не делает ничего (хоткей
  просто не регистрируется, ошибок не возникает).

## Общая проверка

Все файлы проверены на синтаксис (`py_compile`). Логика отдельных
флагов очистки, защита от Zip Slip, виджеты страницы мониторинга и
вся логика игрового режима протестированы изолированно через моки —
все тесты прошли успешно. UI-код (PyQt5) не запускался живьём в
реальном окне, так как тестовая среда без Windows/PyQt5-дисплея —
рекомендую один раз пройтись по страницам «Очистка», «Обновления»,
«Мониторинг» и «Игровой режим» глазами после установки, и на странице
«Игровой режим» проставить свои пути к .exe-файлам (по умолчанию они
пустые — программы будут только закрываться, но не запускаться
обратно, пока пути не указаны).

Новые функции 2026 года (раздел 7) протестированы тем же образом —
изолированной логикой и моками PyQt5/psutil. Регистрация глобального
хоткея (`RegisterHotKey`) и изменения реестра (`winreg` в
`modules/security.py`) специфичны для Windows и не могут быть
выполнены в этой среде — на Windows стоит один раз проверить, что
хоткей реально регистрируется (если комбинация уже занята другой
программой, приложение просто не зарегистрирует её — без ошибки,
это штатное поведение `RegisterHotKey`).
